# 数据快照规范

## 作用

`fetch_redash_dashboard.py` 会先把 Redash dashboard 的 query 结果整理成单个 JSON 快照。AI 生成 HTML 时以这份快照作为唯一输入源。

快照是内部数据准备层，不是最终报告的写作口径。最终 HTML 可以使用快照中的信息做内部判断，并把分析范围、覆盖边界转译成业务语言，不直接展示内部技术统计。

报告生成前，本地必须已经存在一份可用快照作为唯一输入源；这份快照可以是当天复用的现成文件，也可以是本次刚拉取并保存的新文件。

默认路径：

```text
output/school-<school_id>-<YYYY-MM-DD>.json
```

## 顶层结构

快照必须包含以下字段：

- `generated_at`：快照生成时间，ISO 8601 格式。
- `source`：数据来源信息。
- `dashboard`：dashboard 基本信息。
- `fetch_status`：当前增量拉取进度。
- `queries[]`：每个 query 的结构化结果。
- `summary`：拉数成功、缓存命中、补取和失败摘要。

示意：

```json
{
  "generated_at": "2026-04-16T15:20:31+08:00",
  "source": {
    "base_url": "https://redash.seiue.com",
    "dashboard_id": "12",
    "school_id": 539
  },
  "dashboard": {
    "id": 12,
    "name": "希悦使用情况",
    "slug": "usage-dashboard",
    "widget_count": 66,
    "query_count": 24
  },
  "fetch_status": {
    "is_complete": false,
    "started_at": "2026-04-16T15:20:31+08:00",
    "updated_at": "2026-04-16T15:21:08+08:00",
    "total_queries": 24,
    "completed_query_count": 6,
    "pending_query_count": 18,
    "pending_queries": [
      {
        "query_id": 88,
        "query_name": "每周课程任务布置数"
      }
    ]
  },
  "queries": [
    {
      "query_id": 40,
      "query_name": "用户信息",
      "status": "cached",
      "fetch_mode": "cached",
      "columns": [],
      "row_count": 3,
      "rows": []
    }
  ],
  "summary": {
    "total_queries": 24,
    "completed_query_count": 6,
    "success_count": 5,
    "usable_query_count": 4,
    "cached_count": 3,
    "refreshed_count": 1,
    "empty_count": 1,
    "error_count": 1,
    "pending_count": 18
  }
}
```

## `source`

- `base_url`：Redash 站点地址，不包含 API key。
- `dashboard_id`：本次使用的 dashboard id。
- `school_id`：本次请求的学校 ID。

## `dashboard`

- `id`：dashboard id。
- `name`：dashboard 名称。
- `slug`：dashboard slug。
- `widget_count`：dashboard widget 数量。
- `query_count`：去重后的 query 数量。

## `queries[]`

每个 query 至少应包含：

- `query_id`
- `query_name`
- `status`
- `fetch_mode`
- `query_result_id`
- `retrieved_at`
- `columns`
- `row_count`
- `rows`
- `error`

### 状态口径

- `cached`：先前已有可用结果，本次直接命中缓存。
- `refreshed`：缓存不可用，本次仅对该 query 单独补取。
- `empty`：query 成功返回但 0 行，不进入报告正文。
- `error`：query 获取失败，不进入报告正文。

`fetch_mode` 用于记录实际获取方式：

- `cached`
- `refreshed`
- 失败时为 `null`

## `fetch_status`

- `is_complete`：当前快照是否已完成全部 query 拉取。
- `started_at`：本次拉数开始时间。
- `updated_at`：本次快照最近一次写回时间。
- `total_queries`：dashboard 去重后的 query 总数。
- `completed_query_count`：当前已写入快照的 query 数量。
- `pending_query_count`：尚未完成的 query 数量。
- `pending_queries`：尚未完成的 query 列表，便于判断还有哪些模块未到齐。

## `summary`

- `total_queries`：dashboard 去重后的 query 总数。
- `completed_query_count`：当前已完成的 query 数量。
- `success_count`：未报错的 query 数量，包含 `empty`。
- `usable_query_count`：可直接进入正文分析的 query 数量，通常为 `cached + refreshed`。
- `cached_count`：命中缓存数量。
- `refreshed_count`：补取数量。
- `empty_count`：空结果数量。
- `error_count`：失败数量。
- `pending_count`：尚未完成的 query 数量。

## 报告生成使用规则

- 只把 `status` 为 `cached` 或 `refreshed` 且有有效指标的数据集送入正文分析。
- `empty` 和 `error` 不进入目录、正文、图表和表格。
- 如果 `fetch_status.completed_query_count=0`，说明还没有任何可分析数据落盘，此时先等待快照继续更新，不生成空报告。
- 如果 `fetch_status.is_complete=false`，可先基于当前已完成 query 生成阶段性预览，但只能分析已经落盘的模块，不推断尚未完成的模块。
- 如果快照中当前已完成的 query 全部都是 `empty` 或 `error`，停止生成 HTML。
- 如果 `fetch_status.is_complete=true`，应再基于完整快照统一收敛结论和管理建议。
- `summary` 只用于内部判断分析边界、挑选可写模块和识别风险，不作为最终 HTML 的系统提示条或统计摘要原样输出。
- 如需向读者交代分析范围，应改写成自然的分析师口径，例如“本次重点聚焦已有稳定使用记录的核心模块”，并用业务语言概括覆盖边界。
