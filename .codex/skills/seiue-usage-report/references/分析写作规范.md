# 分析写作规范

## 写作框架

每个业务分析段落尽量包含四个部分：

- 判断：先给出管理层能理解的结论。
- 证据：引用具体指标、排名、占比、趋势或表格。
- 可能原因：结合 K12 教务场景解释现象。
- 管理含义：指出对排课、教学运行、班级管理、家校沟通或校内流程的影响。

## K12 场景解释方向

排课与调代课：
- 期初、期末和新学期前后通常是排课使用高峰。
- 调代课低活跃可能提示线下流程仍占主导，也可能说明校内调课需求少，需要结合管理制度判断。

成绩与学情：
- 成绩分析高活跃通常与考试节点、年级质量分析和备课组复盘相关。
- 如果成绩类功能使用不足，应关注数据录入链路、教师分析习惯和年级组应用要求。

考勤与常规管理：
- 考勤稳定使用说明日常管理流程较线上化。
- 如果仅少数角色使用，应关注班主任、年级主任和德育管理岗位的职责分工。

家校沟通：
- 家校沟通活跃度受通知、作业、请假、反馈等场景影响。
- 使用波动需要结合学期活动、考试安排、节假日和学校管理要求解释。

教研与资源：
- 教研模块低使用不一定代表教研薄弱，可能是线下教研或其他平台承载。
- 建议使用“需要核实承载渠道”而不是直接否定管理水平。

## 语言要求

- 使用自然、专业、面向管理层的中文。
- 报告说明和分析范围说明必须像教务分析师交付客户的正式报告，保持业务语言和客户交付口吻。
- 避免模板腔，如“综上所述，该模块表现良好”。
- 避免只描述“上升”“下降”“占比高”“排名靠前”。
- 对缺少证据的原因使用审慎表达，如“可能”“提示”“需要结合校内流程进一步核实”。
- 重点结论可以在 HTML 中用 `<strong>` 或 `<mark>` 标注。

## 禁止写法

- 禁止把相关性写成确定因果。
- 禁止根据使用低就直接判断教师不配合或学校管理不到位。
- 禁止编造学校名称、学生规模、教师规模、考试安排或管理制度。
- 禁止输出“由于没有数据，所以无法分析”这类粗糙结论；应说明可分析范围和补充数据建议。
- 禁止直接暴露数据平台、快照层或拉数流程的系统提示。
- 禁止把学校 ID 的系统格式、内部统计字段或技术状态术语原样带入最终 HTML。

## 建议写法示例

示例：

“排课模块在统计期内使用频次明显集中，符合学校在学期初集中完成教学运行安排的规律。若调代课子模块同期活跃度偏低，则提示日常课务变更可能仍部分依赖线下沟通，建议学校核查年级组、教务处与任课教师之间的线上审批链路是否完整。”

“本次报告重点观察了该校近期已有明确使用记录的核心教务模块，包括活跃趋势、课程教务、考勤请假与家校协同等场景，旨在从日常使用节奏中识别当前线上化推进的重点与改进空间。”
