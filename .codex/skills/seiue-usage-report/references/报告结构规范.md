# 报告结构规范

## 基本结构

报告必须是正式分析报告，不是图表合集。JSON 快照中的 query 数量、返回行数和模块覆盖每次可能不同，章节应根据本次有有效数据的模块动态生成。推荐结构如下：

1. 标题：`XX学校- 希悦系统模块使用深度洞察报告`
2. 报告说明：统计范围、分析范围说明、生成日期和必要口径说明。
3. 学校概览：用户规模、绑定情况、年级结构或学期信息；没有有效数据时跳过。
4. 活跃趋势：日活、周活、角色结构；没有有效数据时跳过。
5. 模块使用分析：按课程教务、成绩评价、考勤请假、家校通知、德育与其他等模块族动态展开；模块族只作为分组标题，组内每个业务模块仍必须单独展开。
6. 薄弱环节诊断：只针对本次数据有证据支持的问题。
7. 专家管理建议：给出可执行的管理动作。
8. 附录或明细表（可选）：只在本次确有必要交付表格时展示必要指标表。

## 内容组织

- 章节必须根据 Redash 返回的有效数据动态增减，不能因为某个 query 无数据就报错。
- 每份报告至少应包含“报告说明”和“管理建议”；如果全部 query 均无有效数据，则停止生成报告。
- 可选模块没有有效数据时跳过，不进入目录、不进入正文、不生成空图表、不生成空表格、不写占位段落。
- “数据覆盖概览”不再作为独立章节输出；分析范围说明应并入“报告说明”，并使用教务分析师口吻概述本次重点覆盖的业务模块与统计范围。
- 每个核心模块必须先判断业务类型，再在“标题说明 -> 图表 -> 分析文字”“标题说明 -> 表格 -> 分析文字”“标题说明 -> 图表 + 表格 -> 分析文字”三种结构里选择其一。
- 每个业务模块必须单独占据一个内容单元，至少拥有独立标题说明，并在需要时配独立图表或独立表格；禁止把两个或以上模块合并成同一张图或同一段分析。
- 趋势解释型、过程节奏型、角色变化型模块默认优先“图表 + 分析文字”，不要机械追加表格。
- 精确交付型、名单长文本型、TopN 排名复核型、绑定率/完成率明细型模块才优先考虑补表。
- 图表不要与分析文字做拥挤的横向并排。
- 学校概览中的绑定率表、年级结构表、学期表等短表默认采用纵向顺序排布，不要为了版面整齐强行做双列并排；尤其在报告开头，应优先保证留白、阅读节奏和信息呼吸感。
- 数据表格应服务于结论，不要无选择地倾倒全部原始行。
- 如果表格不能提供图表和文字之外的新信息，就不要生成该表格。
- 图表编排必须根据该模块的数据结构单独设计，不要让整份报告出现“所有模块都用同一种图表”的模板感。
- 一个 `.figure-block` 只服务一个业务模块，不得在同一图表块中混放多个模块的标题、序列或结论。
- 当一个模块同时有时间趋势、角色结构和总量变化时，优先使用组合图、叠加图或双轴图提升信息密度，而不是简单重复多个普通折线图。
- 如果数据点较多、类目较长或系列较多，优先通过加宽图表、拆分分面、旋转标签、改用横向条形图或附补充表格来保证完整可读，不得出现被遮挡的数据。
- 不能为了让版面整齐而裁掉图表右侧末端数据、压掉图例或省略后段时间点；读图完整性优先于版面平均分配。
- 图例必须完整展示所有核心系列，且位置要服从读图效率，不得因为版面拥挤省略关键图例。
- 如果不同 query 的统计口径互相矛盾，先确认口径，不要强行合并。

## 报告说明口径

- 报告说明应写成教务分析师的自然交付口吻，不要写成系统日志、数据平台说明或脚本运行摘要。
- 可以写明本次分析基于学校阶段性真实业务使用记录整理形成，但不要出现 `Redash API`、`JSON 快照`、`query`、`缓存命中`、`补取`、`失败` 等技术术语。
- 应在报告说明中自然交代本次重点观察的业务范围，例如课程教务、活跃趋势、考勤请假、家校协同等已有明确数据支撑的模块。
- 生成时间可以保留为报告出具时间，但不要与技术快照时间并列输出成系统提示条。
- 如果学校名称无法从查询结果中可靠识别，正文优先使用“该校”“本校”或自然化学校标识；避免出现 `school_id=589` 这类系统格式。
- 不得在报告中展示或暗示 API key。
- 不得原样输出“本次快照共覆盖 55 个 query，其中可用 40 个，缓存命中 54 个，补取 0 个，空结果 14 个，失败 1 个”这类系统统计说明。

## 个性化要求

- 报告必须引用学校名称或自然化学校标识、统计周期、模块名称和关键指标。
- 不要输出通用模板段落。
- 管理建议要对应数据中暴露的问题，例如培训对象、流程节点、模块推广顺序、校内制度衔接。
- 对希悦系统的解释应围绕教务管理实际场景，而不是软件功能说明书。
- 对本次无有效数据的模块，不要写管理结论。

## PDF 风格参考

- 参考“宁波赫德实验学校”PDF 的章节化正式报告感，但不要逐像素复刻。
- 适合采用 A4 竖版阅读节奏、白底、编号标题、正文 bullet、图表块与业务解释交替出现的形式。
- 每次应根据真实数据调整章节数量、图表密度和文字篇幅。

## 输出约束

- 默认输出单文件 HTML 到项目根目录 `output/`。
- 默认先生成并读取项目根目录 `output/` 下的 JSON 快照，再生成 HTML。
- 默认回复输出路径和简短说明，不直接粘贴完整 HTML。
- 没有必要的表格时，可以完全不生成“附录/明细表”章节。
- 只有用户明确要求“直接输出 HTML 代码”时，才在对话中输出完整 HTML。
- 不输出中间思考过程。
