# 模块图表选型规范

## 使用原则

本规范基于 `/Users/alonair/Desktop/Charts/scripts` 下的真实图表脚本归纳，不是抽象的图表教科书。生成报告时，必须先判断模块的数据结构，再决定：

- 该模块是否必须出图
- 该模块适合哪一种图
- 是否必须同时配明细表
- 是否应该只出表不出图

先做业务类型判断，再决定输出形态：

- 模块边界优先于版面整合。每个业务模块都必须单独判断、单独出图或单独出表，不能因为它们同属“课程教务”或“家校通知”就合并展示。

- 趋势解释型：目标是解释波动、峰值、拐点、节奏。默认 `图表 + 分析`，一般不配表。
- 结构对比型：目标是比较角色、状态、类型构成。默认 `图表 + 分析`；只有名称长、TopN 或需要复核精确值时补表。
- 精确交付型：目标是交付绑定率、完成率、名单、学期边界、汇总值。默认 `表格 + 分析`，必要时再加小图。
- 混合型：同时要讲趋势又要交付精确 TopN/明细。采用 `图表 + 表格 + 分析`，但表格只保留关键字段和必要行。

默认规则：

- 有明确时间序列且需要看波动、峰值、拐点、学期节奏时，优先出图。
- 有多个角色、状态、插件或子类型，需要比较结构和变化时，优先出组合图、堆叠图或多序列折线图。
- 类目很少但指标重要时，可“图 + 小表”同时保留，但先确认小表是否真的提供了图表没有交付的精确值。
- 类目很多、名称很长、排序关系比趋势更重要时，优先“排名图 + TopN 表”。
- 只有 1 到 3 行汇总值、绑定率、名单或时间轴信息时，优先表格或信息块，不强行画图。
- 默认不要给每个业务模块追加表格；只有表格能提供图表和正文之外的新增价值时才生成。

## 表格自动决策流程

为每个模块按下面顺序判断：

1. 该模块的主要目标是解释趋势，还是交付精确明细？
2. 如果只保留图表和分析文字，管理层是否已经能完成判断？
3. 表格是否提供以下任一额外价值：精确数值复核、长名称完整展示、TopN 明细、完成率/绑定率/名单交付、关键汇总存档？
4. 如果答案都是否，则不要输出表格。
5. 即使多个模块都落在同一模块族，也必须分别生成自己的图表配置和分析段落，不得合并成“模块组合图”。

输出形态决策：

- `chart_only`：时间趋势、流程节奏、角色结构、状态变化类模块。
- `table_only`：学校概览、学期边界、绑定率、少量汇总值、名单明细、短时间轴信息。
- `chart_plus_table`：既要讲趋势/结构，又需要精确 TopN 或关键明细复核的模块。

## 学校概览

### 用户信息

- 推荐图表：按角色的横向条形图。
- 是否配表格：需要。表格应展示角色人数、绑定率、关键占比。
- 只出表的情况：角色数量很少且重点在绑定率、联系方式覆盖率时，可只出表格和指标卡。
- 原因：该模块重点是规模与结构，图表用于快速比较角色体量，表格用于保留精确人数和绑定率。

### 每年级学生人数

- 推荐图表：不使用图表。
- 是否配表格：建议配简表。

### 最近四个正式学期

- 推荐图表：时间轴图。
- 是否配表格：建议配简表，列出学期名称、开始日期、结束日期。
- 只出表的情况：学期数量极少且页面已很拥挤时。
- 原因：时间轴适合表达学期边界，表格适合准确交付时间信息。

## 活跃趋势

### 日活、周活

- 推荐图表：只需要周活的多角色叠加折线图或多序列折线图。
- 是否配表格：不需要表格。
- 只出表的情况：没有连续时间趋势，只有几个孤立统计点时。
- 原因：核心是比较不同角色在同一时间轴上的活跃变化，图比表更有解释力。

## 课程教务

### 每周课程任务布置数

- 推荐图表：多类型叠加折线图，必要时加合计虚线。
- 是否配表格：可选；只有当需要交付各类型累计量、峰值周、末期值等精确复核值时再配摘要表。
- 只出表的情况：类型过多且时间跨度很短，无法形成趋势解释时。
- 原因：脚本实际采用多序列折线 + 合计线，适合表达任务布置节奏和类型差异。

### 每周课程任务提交与批阅数

- 推荐图表：状态叠加折线图或“多状态折线 + 合计虚线”。
- 是否配表格：可选；只有当需要交付已批阅、已提交、未提交累计值和关键周精确明细时再配摘要表。
- 只出表的情况：只有极少时间点时。
- 原因：重点是比较各状态随时间的同步变化，不应拆成多张单图。

### 每周课表查询次数

- 推荐图表：多类型叠加折线图 + 合计虚线。
- 是否配表格：不需要表格。

### 每周调代课次数

- 推荐图表：按类型的多序列折线图；如果类型较多，可改堆叠柱 + 合计线。
- 是否配表格：不需要表格。
- 原因：重点是日常课务变更节奏和类型结构。

### 每周排课操作数

- 推荐图表：单折线图；如有角色或类型拆分，改多序列折线图。
- 是否配表格：不需要表格。
- 原因：核心是看排课周期峰值与操作节奏。

### 每月选课志愿数

- 推荐图表：按名称或活动的多序列折线图；活动很多时改 TopN 折线 + 其余汇总。
- 是否配表格：建议配 TopN 表。
- 原因：既有趋势，又可能有多个活动并行。

### 选课完成情况

- 推荐图表：排序条形图、完成度对比图或学生数量 vs 完成度组合图。
- 是否配表格：需要。
- 原因：该模块本质是覆盖率和完成度，不仅要看图，还要保留精确分组值。

### 每月课时统计导出次数
- 不需要此模块。
- 推荐图表：单柱状图或单折线图。
- 是否配表格：可选；如果月份少于 6 个，建议直接配表。
- 原因：单指标月度趋势较简单，图表不必过度复杂。

## 成绩评价

### 每周过评项创建数、发布数、过评分录入数

- 推荐图表：
  - 单一指标：单折线图。
  - 创建/发布/录入同时存在：优先多序列折线图或双轴组合图。
- 是否配表格：不需要表格。
- 原因：重点是过程性评价建设的节奏。

### 考试创建数

- 推荐图表：按考场状态的堆叠柱状图 + 合计折线。
- 是否配表格：不需要表格。
- 原因：脚本已采用该结构，适合同时看月度总量和状态构成。

### 教师考核分数录入数、类似“总量 + 用户数/覆盖率”模块

- 推荐图表：双轴柱线图。
- 是否配表格：不需要表格。
- 原因：一条轴放总量，一条轴放人数/覆盖率，更适合管理层读图。

### 每月评教提交人次、每月学期评语采集数

- 推荐图表：按问卷/任务的多序列折线图；任务很少时可用分组柱状图。
- 是否配表格：不需要表格。
- 原因：要同时看时间节奏和任务差异。

### 近半年评价项名称抽样

- 推荐图表：TopN 横向条形图。
- 是否配表格：必须配表，列评价项名称、次数、占比。
- 原因：名称长、类别多，图只能表达排序，表格负责完整文本。

## 考勤请假

### 每周学生请假数

- 推荐图表：按申请类型的堆叠柱状图 + 合计虚线。
- 是否配表格：不需要表格。
- 原因：既要看总请假量，也要看申请类型结构，这类场景堆叠柱优于多条线。

### 每周行政班考勤班次、每周已考勤课节数

- 推荐图表：单折线图。
- 是否配表格：不需要表格。
- 原因：核心是看常规管理的稳定性与异常断点。

### 每周教师打卡数

- 推荐图表：单折线图；如果同时区分角色或打卡类型，改多序列折线图。
- 是否配表格：通常不需要；如果只有 3 到 5 个周点，可补简表。
- 原因：核心是看教师日常管理动作是否形成稳定线上习惯，可与请假、考勤一起判断常规管理线上化程度。

## 家校通知与校务协同

### 每周通知发送量、审批流发起数

- 推荐图表：多指标折线图、按插件堆叠柱图或堆叠柱 + 合计线。
- 是否配表格：不需要表格。
- 原因：通常既有时间趋势，也有渠道或插件结构。

### 每月报表发布人次、查看人次、自定义报告单相关

- 推荐图表：
  - 单指标：柱状图或折线图。
  - 按发布名拆分：多序列折线图或 TopN 组合图。
- 是否配表格：建议配 TopN 表。
- 原因：名称维度多、时间点少时，表格能补精确值。

## 德育、评价与其他

### 每周德育分数录入数、宿舍分数录入数、问卷提交数、听评课人次

- 推荐图表：单折线图。
- 是否配表格：通常不必，除非需要列异常周或分类型明细。
- 原因：这类模块多为单指标时间序列，趋势优先。

### 出入校记录数、约谈人次、心理上报/测评/咨询类

- 推荐图表：按统计粒度选择单折线图或单柱状图；如果同一 query 同时包含多个干预类型，改多序列折线图或分组柱状图。
- 是否配表格：可选；当类型名较长、时间点较少或需要交付精确明细时补表。
- 原因：这类模块核心是看日常管理动作是否连续发生、是否出现明显峰值或断档，不需要过度复杂的图形。

### 每月小红花颁发数量、场馆预约次数、校历事件数

- 推荐图表：
  - 单指标：柱状图。
  - 同时带人数/覆盖率：双轴柱线图。
  - 按名称或类型分解：多序列折线图或堆叠柱图。
- 是否配表格：建议配简表，尤其是名称型维度。
- 原因：月度数据点少，柱状图更适合表达量级；若有拆分维度，再补结构图。

## 何时必须显示表格

出现以下情况时，图表之外必须补表格：

- 需要保留精确人数、绑定率、完成率、占比。
- 需要展示 TopN 排名及具体数值。
- 需要向学校交付可复核的关键汇总指标。
- 图表无法完整承载长名称、名单文本或需要逐项核对的明细。

## 何时可以不显示图表，只显示表格

满足以下任一情况时，可只显示表格或指标块：

- 数据只有 1 到 3 行汇总值。
- 没有时间维度，也没有足够的类目差异。
- 画图后信息量反而低于表格，例如只有几项绑定率、几行学期信息、几行学校概览。
- 图表会因为名称过长、类目过多而严重拥挤，且该模块的核心目标是精确交付而不是趋势解释。

## 何时同时显示图表和表格

以下场景通常建议“图 + 表”并行：

- 用户信息这类既要看结构又要交付绑定率明细的模块。
- 选课完成情况这类既要看覆盖率对比又要保留精确完成值的模块。
- 名称型 TopN 排名模块，例如评价项名称抽样、报表/活动名称排序。
- 时间趋势明确，但管理层仍需要关键峰值周、累计值或末期值做复核的模块。

原因：

- 图表负责讲趋势、结构和对比。
- 表格只负责保留图表未高效交付的精确值、长名称和可复核明细。
- 无论是否同时出图和出表，都只能对应单个业务模块；不要做跨模块拼图式展示。
