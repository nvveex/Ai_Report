# 希悦使用情况报告

## 目标

生成希悦使用情况报告时，用户一旦明确 `school_id`，第一优先级就是先让本地存在一份该学校当天可用的数据快照，再继续后续分析与单文件 HTML 组装。若 `output/` 下已存在该学校当天且结构完整的快照，则直接复用，这也视为“数据已拉到本地保存”；若当天快照不存在、损坏、不匹配，或用户明确要求刷新，再用确定性脚本从 Redash dashboard 拉取并保存新的当天快照。拉数脚本应先初始化快照，再随着 query 完成持续增量更新；HTML、内联 CSS 和 ECharts 配置直接写入同一个 `.html` 文件。报告按本次快照中实际有有效数据的模块动态组织章节，不输出空模块、空图表、空表格，也不只堆叠图表或复述数据升降。

## 默认输出

- 数据快照：`output/school-<school_id>-<YYYY-MM-DD>.json`
- HTML 报告：`output/<学校名或school_id>-希悦系统模块使用深度洞察报告-<YYYY-MM-DD>-<HHMMSS>.html`

项目根目录必须理解为当前包含 `.codex/skills/seiue-usage-report/` 的 Git 仓库根目录，不是 skill 子目录，不是 Codex 临时 worktree 路径，也不是任意当前 shell 工作目录。

- 应优先从当前 `SKILL.md` 所在目录向上寻找仓库根目录，例如通过 `.git/`、仓库根目录 `.env`、根目录 `output/` 或 `.codex/skills/seiue-usage-report/` 辅助判断。
- 默认只写入当前仓库根目录 `output/`；不存在时创建。
- 如果无法定位仓库根目录，先向用户确认项目根目录，不要猜测写入位置。

## 环境配置

生成报告前必须先定位当前仓库根目录，并按以下顺序读取配置：

1. 先读取当前进程环境变量。
2. 如果 `REDASH_API_KEY`、`REDASH_BASE_URL`、`REDASH_DASHBOARD_ID` 任一缺失，则继续读取当前仓库根目录 `.env`。
3. `.env` 只作为本地补充配置来源，不提交到 Git；读取后仅用于本次运行。
4. 如果进程环境和 `.env` 里仍有缺失项，停止并提示用户补齐缺失变量。

必须读取以下配置：

- `REDASH_API_KEY`
- `REDASH_BASE_URL`
- `REDASH_DASHBOARD_ID`

当前仓库 `.env.example` 的默认 dashboard 配置是 `REDASH_DASHBOARD_ID=12`，样例快照常见 dashboard 名为 `单校活跃度看板`；实际运行时始终以本次环境变量、`.env` 或用户明确指定的快照内容为准。

安全要求：

- 不得把 API key 写入仓库、JSON 快照、报告 HTML、日志摘要、临时说明或最终回复。
- `school_id` 必须由用户在本次生成指令中明确提供，例如 `school_id=539`。
- 如果用户未提供 `school_id`，先询问用户提供学校 ID，不要生成报告，也不要使用默认值。

`.env` 读取要求：

- 默认查找位置是当前仓库根目录下的 `.env`。
- 只解析简单的 `KEY=VALUE` 行，忽略空行和以 `#` 开头的注释行。
- 兼容 `export KEY=VALUE` 写法。
- 允许值使用单引号或双引号包裹；读取时去掉外层引号。
- 如果 `.env` 文件存在但编码异常、格式明显错误或关键项为空，提示用户检查 `.env` 内容。

## 快照复用规则

- 用户明确给出 `output/*.json` 路径，或明确要求“只分析现有快照”“只补写/改写 HTML”时，直接读取该快照，不重新访问 Redash。
- 若用户未指定快照路径，则在 `school_id` 明确后，先优先查找 `output/school-<school_id>-<YYYY-MM-DD>.json` 这种“该学校当天快照”；存在且结构完整时默认直接复用，不重新拉数，这一步就视为本地快照已就绪。
- 复用现成快照前，先检查顶层 `source`、`dashboard`、`queries`、`summary` 是否完整；若用户同时给出 `school_id` 或 `dashboard_id`，再校验是否匹配。
- 如果快照存在且 `fetch_status.is_complete=false`，说明这是进行中的增量快照。用户强调“尽快出稿”“先看已有模块”时，可以直接基于当前已完成的 query 生成阶段性预览；若用户要正式定稿，则继续等待或刷新到完整快照。
- 只有当用户明确要求“刷新最新数据”“重新拉取”“强制重新获取”时，才忽略当天快照并重新运行拉数脚本。
- 一份报告只能基于一个 JSON 快照；除非用户明确要求，否则不要把多个学校、多个 dashboard 或多个 snapshot 混合进同一份报告。
- 读取快照后先看 `summary`。`usable_query_count` 明显小于 `total_queries` 在这个看板里是常态；大量 `empty` 通常代表该校该模块暂无数据，不应直接判断为系统异常。

## 执行流程

1. 解析当前仓库根目录，确认或创建根目录 `output/`。
2. 从用户指令中读取 `school_id`；如果既未指定快照又未提供 `school_id`，先追问。
3. `school_id` 一旦明确，先处理“本地快照是否已就绪”这一步，再进入分析和 HTML 生成。
4. 如果用户未指定快照路径，则先查找 `output/school-<school_id>-<YYYY-MM-DD>.json`；存在且结构完整时直接复用，这就视为本地快照已保存。
5. 只有在当天快照不存在、损坏、关键信息不匹配，或用户明确要求刷新时，才继续读取 Redash 配置并运行拉数脚本，把该学校当天快照拉到本地保存。
6. 需要刷新快照时，合并读取进程环境和仓库根目录 `.env`，确认 Redash 配置完整。
7. 需要刷新快照时运行脚本：`python3 .codex/skills/seiue-usage-report/scripts/fetch_redash_dashboard.py <school_id>`。
8. 脚本先读取 dashboard 元数据，提取全部唯一 query，并立即初始化当天快照文件。
9. 对每个 query 采用“缓存优先、缺缓存才单独刷新”策略：先尝试缓存结果；只有没有可用缓存时，才对该 query 单独调用 `POST /api/queries/{query_id}/results` 且使用 `max_age: 0`。
10. 脚本不中断整个 dashboard 拉取：单个 query 失败时写入状态 `error`，继续处理其他 query。
11. 每完成一个 query，就把当前 `queries[]`、`summary` 和 `fetch_status` 原子写回同一路径快照，供后续分析直接读取。
12. 本地快照一旦就绪，报告阶段始终只读取这一份 JSON 快照，不再重新访问 Redash，也不混合多个快照。
13. 先读取快照 `fetch_status`、`summary`、`dashboard` 和 `queries[]`，确认当前已完成 query 数、有效 query 数、失败数、空结果数和覆盖边界。
14. 如果 `fetch_status.completed_query_count=0`，先等待快照继续更新，不生成空报告。
15. 如果快照中当前已完成的 query 全部失败、为空或无法支撑业务判断，停止生成 HTML 并向用户说明。
16. 读取快照中的 `queries[]`，过滤无有效数据的数据集：0 行、核心指标全空、核心数值全为 0、或无法支撑业务判断时不进入报告正文。
17. 先按 query name、列名和样例值识别业务主题，再决定报告章节；字段和模块族规则见 [字段识别规则](references/字段识别规则.md)。模块族只用于归类目录，不可把多个实际业务模块合并成同一个图表或同一个分析区块。
18. 只分析当前已有数据支撑的模块；尚未完成或无数据的模块不进入目录、不进入正文、不生成占位说明。
19. 如果 `fetch_status.is_complete=false` 且用户强调尽快出稿，可先生成阶段性预览 HTML；预览稿只覆盖当前已完成模块，不能把未完成模块写成结论，也不能假设整体覆盖已经稳定。
20. 如果 `fetch_status.is_complete=true`，或用户要求正式定稿，则基于完整快照补齐模块、收敛结论并输出最终 HTML。
21. 基于 K12 教务场景写分析，写作规范见 [分析写作规范](references/分析写作规范.md)。
22. 生成图表前，先按 [模块图表选型规范](references/模块图表选型规范.md) 判断每个模块适合哪种图、是否需要表格、是否只保留表格；先做“业务类型 -> 输出形态”判断，再决定是“图表 + 分析”“表格 + 分析”还是“图表 + 表格 + 分析”。
23. 每个有数据的业务模块必须单独生成章节或子章节，单独设置图表容器、ECharts option 和分析文字；禁止把课程任务、通知、考勤、评价等不同模块合并进同一张图、同一个 figure-block 或同一个结论段落。
24. 直接基于当前 JSON 快照在目标 HTML 文件中写出正文结构、内联 CSS、ECharts 容器和对应 option；不要创建任何额外的 HTML 生成脚本或中间页面脚本，直接产出最终 `.html` 文件。
25. 生成图表：统一使用 ECharts 生成可嵌入 HTML 的交互式图表，并显式使用 SVG 渲染器，例如 `echarts.init(dom, null, { renderer: 'svg' })`；保持 SVG 渲染路径，不使用 canvas。
26. 按 [报告结构规范](references/报告结构规范.md) 组装完整 HTML，并按 [页面样式规范](references/页面样式规范.md) 内嵌 CSS。
27. HTML 文件名必须包含学校名称或学校标识、日期和具体时间，例如 `某学校-希悦系统模块使用深度洞察报告-2026-04-18-153045.html`，确保同一天多次生成不会覆盖历史报告。
28. 默认把单文件 HTML 写入当前仓库根目录 `output/`，最后只回复快照路径、报告路径和必要说明；只有用户明确要求时才在对话中直接输出完整 HTML。

## 拉数脚本契约

脚本路径：

```bash
.codex/skills/seiue-usage-report/scripts/fetch_redash_dashboard.py
```

默认调用：

```bash
python3 .codex/skills/seiue-usage-report/scripts/fetch_redash_dashboard.py 539
```

脚本输入：

- 进程环境或仓库 `.env` 中的 `REDASH_API_KEY`
- 进程环境或仓库 `.env` 中的 `REDASH_BASE_URL`
- 进程环境或仓库 `.env` 中的 `REDASH_DASHBOARD_ID`
- 必填 `school_id`
- 可选 `--output`

脚本输出固定为 JSON 快照，结构见 [数据快照规范](references/数据快照规范.md)。未显式传 `--output` 时，默认输出到 `output/school-<school_id>-<YYYY-MM-DD>.json`，日期按 `Asia/Shanghai` 自然日解释。脚本会先初始化快照，再在每个 query 完成后增量写回同一路径。报告生成阶段把这个快照视为唯一数据输入，并在需要时可基于增量快照先生成预览稿，再在完整快照完成后定稿。

状态口径：

- `cached`：命中 Redash 已有缓存结果。
- `refreshed`：缓存不可用后，仅该 query 被实时补取。
- `empty`：query 成功返回但 0 行。
- `error`：query 获取失败；记录错误摘要但不泄露 API key。

## Redash 专用识别策略

采用“希悦看板优先、查询数量可变、字段容错”的半结构化策略，不把本 skill 当作任意 Redash 看板解析器。

- 当前仓库中的样例通常对应 dashboard `12` / `单校活跃度看板`，但实际运行时始终以本次配置和快照为准。
- 每次以 JSON 快照中实际返回且有有效数据的 query 为准。
- Redash 看板中的 query 数量、模块覆盖和返回行数可以不同。
- 优先按 query name 识别模块、用户、角色、时间和使用指标，再用列名和样例值校验。
- 允许有限同义字段映射，兼容 Redash 查询列名差异。
- 如果学校名称无法可靠识别，可用 `school_id-<ID>` 作为输出文件名前缀；正文优先使用“该校”“本校”或自然化学校标识，不写 `school_id=<ID>` 这种系统格式。
- 如果统计周期或核心字段无法可靠映射，停止并向用户确认字段含义。
- 不要为了生成完整报告而臆造学校名称、模块口径、时间范围或业务原因。

## 分析要求

- 报告应先用自然语言交代本次分析范围，例如“本次重点观察已有明确使用记录的核心教务模块”；本次部分模块暂无有效数据在该看板里通常是常态，但不要把这类内部状态原样写进正文。
- 每个核心判断必须尽量关联到具体指标、表格或图表。
- 每个模块的判断必须只基于该模块自己的图表、表格和指标展开，不要把多个模块揉成同一段综合结论。
- 表格不是每个模块的固定组件；只有当业务类型、信息密度和交付目标确实需要精确明细时才生成表格。
- 正文中的方法说明、范围说明和口径说明必须写成教务分析师语言，不得出现系统提示、学校 ID 的系统格式或内部拉数统计摘要。
- 快照中的 `summary`、`usable_query_count`、`error_count`、`cached_count`、`refreshed_count` 仅用于内部判断分析边界、选择可写模块和控制风险，不得原样写进正文。
- 如果需要说明分析范围，应写成“本次重点观察课程教务、活跃趋势、考勤请假等已有明确使用记录的模块”这类自然表述，不要写技术统计摘要。
- 每个业务段落使用“判断 + 证据 + 可能原因 + 管理含义”的表达方式。
- 必须从真实 K12 教务场景解释“为什么”，例如排课周期、学期节点、班主任使用习惯、教研组管理流程、家校沟通链路等。
- 禁止只写“某模块上升/下降/占比高”。
- 禁止无证据因果断言；原因不足时使用“可能”“提示”“需要结合校内流程进一步核实”等审慎表述。
- 无有效数据的模块不得写入报告，也不得在管理建议中引用。

## 图表要求

- 每个图表都必须根据当前业务数据的字段结构、时间粒度、分类维度和结论目标单独设计，不得为了省事重复套用同一种图表模板。
- 每个业务模块必须拥有独立图表配置；禁止把多个模块的数据系列并入同一个 option，也禁止复用同一个图表容器承载多个模块。
- 优先选择能同时呈现多个业务维度的高级图表，例如分组柱状图、堆叠柱状图、双轴柱线组合图、多序列折线图、分面小图和带总量对比的叠加图。
- 同一模块中如果存在多个角色、多个状态或多个并行口径，且核心目标是比较它们随时间的变化关系，优先使用叠加折线图或多序列折线图，不要拆成多张单线图。
- 只有在数据结构确实适合时才使用环形图；时间趋势、流程节奏、角色对比、任务量与活跃量联动时，优先使用折线图、柱状图或柱线混合图。
- 必须确保全部数据可见、不被遮挡：类目过多时允许横向拉宽图表、旋转类目标签、分面拆图、分组分页或改成明细表，不得让标签、柱体、点位或图例互相压住。
- 右侧最后一个数据点、最后一组柱体、最后一个时间刻度和对应数据标签必须完整显示，不得被画布边界裁切。
- 如果横轴内容过密导致显示不全，优先按以下顺序处理：增加右侧和底部留白、压缩横坐标间距、减小柱间距、减少刻度文字密度、旋转标签、横向拉宽图表；仍拥挤时改为分面图或“主图 + 补充表格”。
- 横轴过密时，优先压缩每个横坐标类目之间的间距和柱间距，不要通过缩小整张图片比例来硬塞内容。
- 可以减少横轴文字标签密度，但不能省略真实数据点本身；所有时间点和业务数据都必须完整绘制。
- 构造图表数据时，不得用 `sorted(points)[:12]`、`points[:12]`、`labels[:12]` 或任何固定长度切片硬截断时间序列或类目序列，除非用户明确要求只展示最近 N 个点。
- 图表必须有标题、完整图例、坐标轴标签和必要的数据标签；图例应放在不遮挡数据的位置，并确保全部系列名称显示完整。
- 折线图必须显示峰值标签；如果同一张图中有多条核心折线，则每条核心折线至少显示一个峰值标签。峰值用 ECharts `markPoint` 呈现，`symbol` 使用水滴形 `pin`，标签内容只显示具体数值，例如 `128`，不附加“峰值”“次”“人”或解释性短语。
- 如果图例文字较多、较长或系列较多，必须主动增加图例项间距、行间距、列间距和图例区域留白，避免图例文字互相压住或被图表主体遮挡。
- 如果数据标签过多导致重叠，折线图至少保留水滴形峰值标签，其余再按末值、异常值和关键节点标签的优先级保留，不得让大面积标签互相覆盖。
- 峰值、末值或异常值标签一旦显示，必须包含具体数值，不得只显示“峰值”“末值”等抽象文字；其中峰值水滴标签只显示数值本身，例如 `394`。
- 配色应年轻化、清爽且有层次，优先使用 `#39B981`、`#265BEA`、`#F59E11`、`#8554F6`、`#3CB6D4` 等较鲜艳但专业的现代商业色，不要使用过于黑暗、发灰或低辨识度的主色。
- 图表服务于论证，不做磁贴式图表汇编。
- 如果数据不支持某类图表，改用表格和文字说明，不输出实现性提示。
- 当一个结论需要同时展示总量、结构和趋势时，不要拆成多个重复图；应优先设计一个信息密度更高但仍清晰的组合图。
- 实现时必须显式调整图表宽度、高度、边距、tick 间距、柱宽、bargap、label padding 和 x 轴范围，保证读图优先于默认库配置。
- 折线图必须严格按真实时间顺序或真实类目顺序绘制，保证 x 值与 y 值一一对应；禁止因错误排序、错误去重、缺失补位错位或坐标映射错误造成折线“漂移”或折点位置偏移。
- 每个图表必须用同一个 `.figure-block` 包裹“图表标题 + 图表主体 + 图表解读/数据来源”，并在打印 CSS 中避免该块被分页切开。

## HTML 输出要求

- 输出完整单文件 HTML，包含 `<!doctype html>`、`<html>`、`<head>`、内嵌 CSS 和 `<body>`。
- 直接生成最终 HTML 文件，不创建额外的 HTML 生成脚本、页面模板脚本或中间渲染脚本。
- 必须使用内联 CSS，严格遵循 [页面样式规范](references/页面样式规范.md) 中的顶级咨询研报级排版规范。
- 生成 CSS 前必须先按“平台 + 功能/主题 + 视觉风格 + 颜色/灯光 + 细节/组件”的公式确定样式提示词；默认平台为“单文件 HTML 商业咨询报告”，功能/主题为“希悦使用情况深度洞察”，视觉风格为“顶级咨询研报级极简文档流”，颜色/灯光为“纯白背景、深炭黑正文、沉稳商务色图表”，细节/组件为“三线表、底部实线章节标题、白底图表、斜体图表解读、Tailwind CSS 和 Lucide 图标”。
- 页面背景必须为 `#FFFFFF`。
- 主容器居中，最大宽度约 `900px`，上下保留文档式留白。
- 默认采用“标题衬线体 + 正文无衬线体”的高对比度排版。
- 标题示例：`XX学校- 希悦系统模块使用深度洞察报告`。
- 必须包含“报告说明”和“管理建议”，其他章节根据实际数据覆盖动态生成。
- “分析范围说明”必须并入“报告说明”，不得再单独输出“数据覆盖概览”这类偏系统说明章节。
- 禁止使用互联网风格的圆角卡片、阴影和大面积色块分区。
- 表格应按业务类型自动决定是否出现，不得为了版式统一给所有模块都附一张表。
- 如果图表和文字已经足以支撑判断，不要重复生成装饰性表格。
- 表格应采用三线表规范，不使用斑马纹和全封闭网格线。
- 学校概览中的短表、信息块和说明文字默认采用纵向文档流排列；除非两块内容在宽度、行数和信息密度上都明显均衡，否则不要把“用户绑定明细”“年级结构”“学期信息”等短表并排放在同一行，以免页面头部显得拥挤。
- 必须加入打印样式，避免图表、短表格、标题和图表解读跨页断裂；图表高度应控制在可打印范围内。
- 重点结论可用 `<strong>` 或 `<mark>` 标注。
- 页面必须响应式适配桌面和移动端。

## 需要先确认的情况

遇到以下情况时，先向用户确认，不要直接生成报告：

- 用户未提供 `school_id`。
- 学校名称、统计时间范围或核心模块口径无法从 JSON 快照中可靠识别。
- 模块名、使用次数、活跃用户、使用时长等核心字段无法映射。
- 查询结果明显是脱敏样例、字段缺失严重或存在互相矛盾的合计口径。
- 用户要求把多个学校、多个 dashboard 或多个 JSON 快照合并成同一份正式报告，但未明确合并口径。
- 用户要求的输出范围与 Redash 实际返回数据范围不一致。
- 全部 query 均无有效数据，无法支撑报告正文。
